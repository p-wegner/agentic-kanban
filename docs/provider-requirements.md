# Agent Provider Requirements

This document records the provider features agentic-kanban depends on and how each supported harness satisfies them. Keep it current when changing `packages/server/src/services/agent-provider/*` or when upgrading provider CLIs.

Local CLI surfaces verified on 2026-05-31:

- `codex exec --help`
- `copilot -h`

## Required Capabilities

| Capability | Why the board needs it |
|---|---|
| Non-interactive prompt execution | Workspaces launch agents automatically from the board. |
| Structured streaming output | The UI parses progress, tool activity, final text, usage, and completion state. |
| Provider session ID and resume | Follow-up turns and restarts must continue the correct provider-local session. |
| Working directory control | Each workspace runs in its own worktree. |
| MCP/tool configuration | Agents need the board MCP tools and file/shell capabilities. |
| Permission controls | Board-launched agents must run without interactive approval prompts. |
| Plan/read-only mode | High-priority issues can run a planning pass before implementation. |
| Initial context delivery | Ticket description, generated context primer, and handoff context must reach the first turn. |

## Compatibility Matrix

| Feature | Claude Code | Codex | Copilot |
|---|---|---|---|
| Launch command | `claude -p` | `codex exec -` | `copilot -p` |
| Structured output | `--output-format stream-json --verbose` | `codex exec --json` | `--output-format json --stream on` |
| Resume | `--resume <id>` | `codex exec resume <id>` | `--resume=<id>` |
| Working directory | Spawn `cwd` | Spawn `cwd` | Spawn `cwd` |
| MCP config | `--mcp-config <file>` | Uses Codex MCP/config support generated by the harness config | `--additional-mcp-config @<file>` plus `--disable-builtin-mcps` |
| Non-interactive permissions | `--dangerously-skip-permissions` via args/settings | `--dangerously-bypass-approvals-and-sandbox` | Allow-list flags, or `--allow-all` when configured |
| Plan mode | Claude permission mode plus appended plan prompt | Read-only sandbox plus plan prompt prefix | `--plan`, restricted tool list, denied write/commit tools |
| Text context file attachment | Loaded implicitly from `CLAUDE.local.md` by Claude Code | Not supported by current local CLI | `--attachment <path>` |
| Context primer delivery | `CLAUDE.local.md` in worktree | `CLAUDE.local.md` contents appended to stdin prompt as fallback | `CLAUDE.local.md` passed as `--attachment` |
| Image attachment | Not wired by board launch path | `--image <file>` exists, not wired | `--attachment <path>` exists, not wired for visual input |

## Initial Context Contract

Workspace creation writes a gitignored `CLAUDE.local.md` file into the worktree root. It contains:

- Ticket title and description.
- Optional generated context primer.
- Any future per-workspace first-turn context that should not enter git history.

Provider handling:

- Claude Code: relies on Claude Code loading `CLAUDE.local.md` as local project memory.
- Copilot: passes the file path through `--attachment`, which the current CLI documents for non-interactive mode.
- Codex: the current local CLI has no repeatable text/document attachment flag. The board reads the file and appends its contents to the stdin prompt so Codex receives equivalent context on the first turn.

## Known Unsupported Or Partial Features

| Gap | Provider(s) | Current behavior |
|---|---|---|
| Native text/document attachment | Codex | Fallback prompt injection is used. Replace with a real file flag if the CLI adds one. |
| Provider-neutral local memory filename | All | The canonical generated file is still named `CLAUDE.local.md` for Claude compatibility. Other providers receive it through launch wiring. |
| Binary/non-text issue attachments as first-turn context | All | Not part of the workspace launch contract yet. |
| Interactive approval prompts | All board launches | Unsupported by design; board sessions must run unattended or fail clearly. |
| Cloud-hosted/multi-user auth flows | All | Out of scope for this local-first app. |

## Update Rules

- Verify provider flags against the installed CLI before changing launch construction.
- Add or update tests in `packages/server/src/__tests__/agent-provider.test.ts` for every provider flag change.
- Keep context delivery equivalent across providers. If a provider lacks native attachment, document and test the fallback.
- Do not assume Claude-specific files are read by Codex or Copilot unless the launch path explicitly bridges them.
